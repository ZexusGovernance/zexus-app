## Суть продукта

**Zexus** — платформа доверия для DeFi-проектов на **Base Mainnet**.

Проблема: в крипте полно проектов, которые обещают дорожную карту и не выполняют. Zexus решает это через:
- публичный **Trust Score** (0–100) для каждого проекта
- **Verdict** — голосование сообщества по каждому milestone
- **Emergency Call** — сигнал тревоги от холдеров
- **ZXP** — внутренний токен, которым мотивируются все участники

## Страницы (навигация слева)

### 1. Feed — лента событий
Реалтайм-поток событий от всех проектов. Посты 4 типов:

| Тип | Цвет | Когда |
|---|---|---|
| **Update** | синий | Обновление прогресса / новость |
| **Verdict** | зелёный | Запрос на подтверждение milestone |
| **Alert** | янтарный | Критическое предупреждение |
| **New** | фиолетовый | Новый проект |

Справа в сайдбаре: активное голосование + "Trust Score movers 24h".

### 2. Projects — каталог проектов
Список верифицированных проектов Base Mainnet. У каждого проекта есть страница с вкладками:
- **Timeline** — история Trust Score
- **Milestones** — дорожная карта
- **Votes** — история Verdict-голосований
- **Holders** — верифицированные холдеры

### 3. Alerts — критические уведомления
Только события из вотчлиста пользователя: пропущенные дедлайны, открытые вердикты, Emergency Calls.

### 4. Predict Market — предсказания
Ставки на внутренние события платформы (достигнет ли Zexus 1000 юзеров, запустится ли on-chain стейкинг и т.д.). Валюта — только ZXP, комиссия 1-2%.

### 5. Staking ZXP — стейкинг
Два уровня:
- **Layer 1** (off-chain) — без газа, мгновенный вход, ~8% APY
- **Layer 2** (on-chain Base) — более высокий APY до 25%

APY плавающий: пересчитывается раз в неделю. Чем больше пул — тем ниже APY. **Genesis-стейкеры** навсегда фиксируют свой множитель.

## Ключевые концепции

### Trust Score (0–110)
Публичный рейтинг доверия к проекту. Кап 110 (в базе), отображается 0–100.

**Начальный скор — рассчитывается в Admin Panel:**

| Категория | Макс | Что учитывается |
|---|---|---|
| Social | 25 | Twitter: ≥20k→+15, ≥5k→+10, ≥1k→+5 · Discord: ≥1k→+5, ≥100→+3 · GitHub commits/30d: ≥50→+5, ≥11→+4, ≥1→+2 |
| Product | 25 | Mainnet→+15, Testnet→+7 · Whitepaper→+3 · Security audit→+12 |
| On-chain | 20 | Wallets: ≥10k→+15, ≥1k→+10, ≥100→+4 · TVL: ≥$1M→+5, ≥$100k→+3 |
| Team | 15 | Doxxed→+8 · Has investors→+7 |
| Track record | 15 | Contract age: ≥12mo→+10, ≥6mo→+5 · Partnerships→+5 |
| Bonus | 10 | CEX listing→+4 · Foundation grant→+3 · Media→+3 · Hackathon→+2 · Top integration→+3 |
| **Итого** | **110** | |

Данные по GitHub и BaseScan подтягиваются автоматически, остальное вводится вручную.

**Бейджи уровня доверия (визуально вместо просто цифры):**

| Диапазон | Цвет | Бейдж |
|---|---|---|
| 0–44 | красный | Unproven |
| 45–69 | жёлтый | Developing |
| 70–89 | зелёный | Trusted |
| 90–110 | золотой | Elite |

Бейдж меняется **мгновенно** при изменении Trust Score — всегда отражает текущее число без задержек.

**Растёт, когда:**
- Кворум набран и Yes победило на Verdict (макс +5)
- Проект закрывает milestone в срок
- Холдеры верифицируются

**Падает, когда:**
- Дедлайн пропущен (автоматически −18 pts)
- Emergency Call подтверждён командой Zexus
- No победило на Verdict (макс −10)
- Verdict аннулирован из-за недобора кворума (штраф −1)

### Milestone → Verdict — как они связаны

Verdict не создаётся проектом вручную как обычный пост. Он **автоматически создаётся** когда проект нажимает "Submit for verification" на конкретном milestone. Это исключает пустые Verdicts ради накрутки поинтов.

**Полная цепочка:**
```
Проект создал milestone с дедлайном
        ↓
Выполнили → нажали "Submit for verification"
        ↓
Система автоматически создаёт Verdict:
  - пост в ленту (тип verdict, зелёный)
  - запись в таблицу verdicts
  - уведомление в Alerts для watchlist
        ↓
72 часа открыто голосование
        ↓
Yes победило + кворум → milestone ✓ confirmed, Trust Score +X
No победило + кворум  → milestone ✗ rejected, Trust Score −X
                         можно переподать через 14 дней или отменить milestone
Кворум не набран      → void, Trust Score −1, кулдаун 14 дней
        ↓
Дедлайн прошёл, Submit не нажали → автопенальти Trust Score −X
```

**Вкладка Votes в профиле проекта** — история всех Verdicts этого проекта (открытые и закрытые). Отдельная сущность не нужна, это та же таблица verdicts.

**Структура таблиц (нужно создать):**
```
milestones:  id, project_id, title, deadline, status
             status: pending | submitted | confirmed | rejected | missed

verdicts:    id, milestone_id, project_id, created_at, closes_at, status
             status: open | closed | void
             yes_count, no_count

votes:       id, verdict_id, voter_wallet, vote (yes/no), created_at
```

### Verdict — кворум и формула Trust Score

**Кто может голосовать:**
- Кошельки с **≥ 5 ZXP застейкано** (Layer 1 или Layer 2)
- Аккаунт должен быть **≥ 48 часов** на момент открытия Verdict
- 1 кошелёк = 1 голос, размер стейка не влияет (защита от китов)

**Кворум:**
```
required = max(5, floor(active_voters_30d × 0.03))
```
`active_voters_30d` — кошельки голосовавшие хотя бы раз за 30 дней. Растёт органически с платформой.

**Формула изменения Trust Score:**
```
decisiveness  = |yes% - 50%| / 50         // 0.0 = ровный split, 1.0 = единогласно
participation = min(2.0, votes / required) // насколько превысили порог
direction     = yes > 50% ? +1 : −1

impact = round(direction × decisiveness × participation × 5)
```
Кап **несимметричный: макс +5, мин −10** — провал бьёт вдвое сильнее.

**Примеры:**
- 80% Yes, 30 голосов, порог 20 → `+1 × 0.6 × 1.5 × 5 → +5`
- 55% Yes, 22 голоса, порог 20 → `+1 × 0.1 × 1.1 × 5 → +1`
- 90% No, 60 голосов, порог 20 → `−1 × 0.8 × 2.0 × 5 → −8`

**Текущий баг:** `app/api/posts/route.ts` применяет `trust_score_change` мгновенно при создании поста — проект сам себе ставит поинты. Нужно убрать поле из формы для Verdict, считать автоматически после закрытия.

### ZXP — получение и право голосовать

**Условие для голосования на Verdict:**
- ≥ 5 ZXP застейкано (Layer 1 или Layer 2)
- Аккаунт существует ≥ 48 часов
- Стейк может быть свежим если аккаунту уже 48+ часов

**Онбординг — день 1:**

| Действие | Награда | Лимит |
|---|---|---|
| Подключить кошелёк впервые | +3 ZXP | 1 раз |
| Первая реакция на пост (лайк/дизлайк) | +1 ZXP | 1 раз |
| Первый комментарий | +2 ZXP | 1 раз |
| Первый watchlist | +1 ZXP | 1 раз |
| **Итого день 1** | **+7 ZXP** | |

Стейкнул 5 ZXP → осталось 2 в кошельке → на 3-й день можно голосовать.

**Регулярные награды:**

| Действие | Награда | Лимит |
|---|---|---|
| Daily check-in | +1 ZXP | 1 раз в день |
| Правильный vote (совпал с большинством) | +2 ZXP | за каждый закрытый Verdict |
| Пригласить юзера который застейкал | +10 ZXP | реферальная программа |

**Защита от фарма (без идентификации личности):**
- Аккаунт ≥48 часов — нельзя создать и сразу голосовать в тот же день
- Кворум считается от `active_voters_30d` — 1000 фейков не помогут без реальной 30-дневной активности
- Правильный vote = +2 ZXP только если совпал с итогом — рандомные клики убыточны
- Реферал считается только при факте стейкинга приглашённого

### Emergency Call

Используется когда проект исчез, не отвечает, явно обманывает или случился инцидент (взлом, rug pull и т.д.).

**Механика — полная цепочка:**
```
Любой пользователь открывает пул Emergency Call
Ставит от 5 до 100 ZXP (локируются)
        ↓
48 часов окно: другие присоединяются, каждый ставит от 5 ZXP
Минимум 3 участника, максимум 50
Цель пула: 300 ZXP
        ↓
Пул не достиг 300 ZXP за 48ч →
  Call отменяется, все получают ZXP обратно (без штрафа)
        ↓
Пул достиг 300 ZXP → Call активирован:
  - критический алерт в ленте и в Alerts всех из watchlist
  - проект уведомлён
  - Trust Score помечается "Under Review"
        ↓
48 часов: проект должен ответить публично (пост в ленту)
        ↓
Команда Zexus проверяет и выносит решение:
```

| Исход | Что происходит |
|---|---|
| Ложная тревога | Все участники теряют ZXP из пула (сжигаются). Trust Score восстанавливается |
| Реальная проблема, проект ответил и исправил | Участники получают ZXP обратно − 1 ZXP каждый (комиссия). Trust Score −10 |
| Реальная проблема, проект проигнорировал | Участники получают ZXP обратно + 3 ZXP бонус каждый. Trust Score −20 |

**Защита от злоупотреблений:**
- Минимум 3 участника — один человек не может атаковать проект
- Пул 300 ZXP — экономически затратно организовать ложный Call
- Финальное решение за командой Zexus
- Один проект не может иметь больше 1 активного Emergency Call одновременно
- ZXP из ложных вызовов сжигаются → дополнительный сток против инфляции

**Что считается валидным ответом проекта:**
- Опубликован публичный пост в ленту (тип Update или отдельный Response)
- Пост явно ссылается на Emergency Call
- Приложены доказательства: ссылки, on-chain транзакции, скриншоты
- Команда Zexus проверяет в течение 72ч после ответа и выносит решение
- Если проект ответил "мы работаем над этим" без доказательств — Zexus решает достаточно ли этого

**Закрытые и неактивные проекты:**
- Проект закрылся → доступ к аккаунту остаётся, вся история видна
- Режим ограниченного доступа: максимум 1 пост в неделю
- Чтобы изменить статус или восстановить полный доступ → написать в поддержку Zexus
- Профиль помечается бейджем "Inactive" если нет активности 90+ дней

**Статус реализации:** логика определена, код не написан.

### ZXP — токономика

**Эмиссия (создание ZXP):**

| Действие | Награда | Лимит |
|---|---|---|
| Подключить кошелёк впервые | +3 ZXP | 1 раз |
| Первая реакция на пост | +1 ZXP | 1 раз |
| Первый комментарий | +2 ZXP | 1 раз |
| Первый watchlist | +1 ZXP | 1 раз |
| Daily check-in | +1 ZXP | 1 раз в день |
| Правильный vote на Verdict | +2 ZXP | за каждый закрытый Verdict |
| Реферал (пригласил юзера который застейкал) | +10 ZXP | без лимита |

Онбординг день 1: **+7 ZXP** → стейкнул 5 → через 48ч право голосовать.

**Сток (уничтожение ZXP — защита от инфляции):**

| Событие | Сколько сжигается |
|---|---|
| Genesis Pass оплата в ZXP | 30 ZXP/месяц |
| Ложный Emergency Call | весь пул участников (300+ ZXP) |
| Комиссия при успешном Emergency Call | 1 ZXP с каждого участника |
| Predict Market комиссия | 1–2% от пула ставок |

**Контроль инфляции:**
Суммарная эмиссия ограничена кап-эмиссией — максимум X ZXP минтится за день на всю платформу. Чем больше юзеров делают check-in, тем меньше получает каждый (не фиксированный +1 а долевой). Точное значение кап-эмиссии определяется при запуске.

**Genesis Pass — двойная цена:**
- **$1 в месяц** (стабильная цена, не зависит от курса ZXP)
- **ИЛИ 30 ZXP в месяц** (сжигаются, создаёт спрос на токен)

Если ZXP инфлирует → люди платят ZXP → много сжигается → инфляция замедляется. Самобалансирующаяся система.

**Статус:** off-chain (Supabase), on-chain Layer 2 на Base — при релизе.

## Бизнес-логика: кто что может делать

| Роль | Действия |
|---|---|
| Гость | Смотреть ленту, проекты, Trust Score |
| Пользователь с кошельком | + Стейкать ZXP, голосовать на Verdict (после 48ч + ≥5 ZXP стейк) |
| Проект (верифицированный адрес) | + Публиковать Update / Alert, создавать milestones, Submit for verification |

**Как проект попадает на платформу:**
1. Заполняет форму заявки
2. Команда Zexus проверяет и анализирует проект
3. Отправляют подтверждение через официальный источник в X или на почту
4. Проект получает доступ к аккаунту — `admin_wallet` прописывается в базе
5. Профиль публикуется с начальным Trust Score

## Уведомления — Telegram Bot

Telegram предпочтительнее email: выше open rate, мгновенная доставка, не попадает в спам.

**Как работает:**
- Пользователь вводит `/start` в боте → получает уникальный код → вводит его на платформе → кошелёк привязан к Telegram аккаунту
- Уведомления на платформе (Alerts) и уведомления в боте — одна система, одни и те же события

**Что бот присылает:**

| Событие | Кому |
|---|---|
| Открылся Verdict для проекта | всем у кого проект в watchlist |
| Emergency Call активирован | всем у кого проект в watchlist |
| Trust Score проекта изменился ±5+ pts | всем у кого проект в watchlist |
| Твой vote закрылся — результат и +ZXP | проголосовавшему |
| Напоминание о daily check-in | всем (опционально, можно отключить) |
| Ответ на Emergency Call от проекта | участникам пула |

**Технически:** отдельный Telegram Bot (Node.js / Python) подключён к Supabase через webhooks или polling. При событии в базе → триггер → бот шлёт сообщение нужным chat_id.

**Статус:** не реализован, делать после основных механик.
